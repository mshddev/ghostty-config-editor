# Pass-2 UX Audit Digest — 2026-07-04

Verified findings from the 8-lens ultracode audit (each lens adversarially verified against source by a second agent), grounded in the firsthand walkthrough of the packaged app (50 observations) and the view-layer source. 89 findings + 9 completeness-critic gaps.

## Visual design system & tokens (DS)

### DS-1 [P1/fix/M] Slider numeric controls show a bare 0-1 float with no unit/% — and on the top-billed "Window transparency" that float is a live semantic inversion
**Evidence:** numeric-specs.json defines 7 `.slider` specs — background-opacity/background-image-opacity/unfocused-split-opacity/bell-audio-volume/cursor-opacity/faint-opacity (all 0-1) plus minimum-contrast (1-21) — none carrying `unit`. OptionListView.swift:1027-1042 `sliderEditor` renders `Slider` + `Text(numberString(...))` and never reads `spec.unit`, unlike `fieldEditor` (1063-1081) which does at 1073-1075. background-opacity is common-tier rank 1 (option-tiers.json:4) labeled "Window transparency" (option-labels.json:7) yet its slider surfaces raw "1" = fully opaque (Obs 11) — 1 reads as "most transparent" but means "not transparent at all."
**Proposal:** Add a display transform to NumericSpec (`unit` + a `displayScale` for 0-1→0-100%) and route every `.slider` through one shared `LabeledSlider` that consults it the way `fieldEditor` already consults `spec.unit`. For the transparency-labeled option specifically, either invert the shown scale (transparency% = 1-opacity) or relabel to "opacity," and add leading/trailing endpoint captions ("Opaque"/"Transparent") so no slider ever shows a bare 0-1 float and the direction is unambiguous.

### DS-2 [P2/fix/S] Color swatch ring uses the low-contrast adaptive `.separator`, so a dark set color and an unset swatch are indistinguishable on a dark card
**Evidence:** The ring exists but is too faint: OptionListView.swift:836 and :876 overlay `.strokeBorder(.separator, lineWidth: 1)`, and the preset grid :908-909 uses `Color(nsColor: .separatorColor)` for the unselected ring. `.separator` is a low-alpha content-divider hairline, so a dark user color renders identically to the unset `.quaternaryLabelColor` fill (:859) — Obs 27: "a dark swatch on a dark card reads as 'nothing there'; unset vs dark-value indistinguishable."
**Proposal:** Replace `.separator` with a dedicated higher-contrast swatch ring — a stacked double-stroke (inner black-alpha + outer white-alpha, like AppKit's own color-well bezel) — applied at all three sites (row :836, editor preview :876, preset grid :908) so 'set to a dark color' and 'unset' are always visually distinct regardless of fill luminance.

### DS-3 [P2/feature/L] No shared design-token layer: spacing, radius and opacity are hand-written per call site across the view layer
**Evidence:** Verified via grep across Views/ + App/: 7 distinct raw cornerRadius values (2/4/5/6/8/10/16) and 9 distinct ad-hoc opacity fills across ~17 call sites (0.04/0.08/0.10/0.12/0.15/0.28/0.35/0.4/0.5) spread over GlobalFindView, KeybindEditorView, OptionListView, SurfaceChrome, ThemeBrowserView, WelcomeView. The only shared metric today is `RowMetrics` in SurfaceChrome.swift:12-19 (row padding + two fonts); radius, opacity/tint scale and spacing rhythm are reinvented per file (also noted as a repo fact in research-context.md).
**Proposal:** Add a `DesignTokens.swift` (or extend SurfaceChrome.swift) with a spacing scale, a radius scale (tight/standard/card = 4/6/10) and a semantic tint scale (subtleFill = `.secondary.opacity(0.10)`, accentFill = `.accentColor.opacity(0.15)`) and migrate the literals onto it. Sequence this AFTER the pill/badge consolidation (separate finding), since that component alone retires most of the duplicate opacity/radius literals.

### DS-4 [P2/refinement/M] Six hand-rolled pill/badge implementations with drifting metrics, and the same 'Customized' state renders title-case on the row but lowercase in the popover
**Evidence:** Six independent capsule pills with different padding/opacity for the same idiom: OptionListView.swift `Badge` (2072-2089: 7h/3v, tint.opacity(0.12)) and `customizedAffordance` (419-441: 7h/2v, accentColor.opacity(0.15)); KeybindEditorView.swift `badge` (549-555: 6h/2v, badgeTint.opacity(0.15)); ThemeBrowserView.swift `currentPill`+`appearanceBadge` (207-227: 6h/1v); GlobalFindView.swift `categoryPill` (164-171: 6h/1v). Wording drift: the row shows `option.state.displayName` = "Customized" (OptionListView.swift:421) but the popover `stateBadge` hardcodes lowercase "customized" (OptionListView.swift:1907), bypassing the shared vocabulary (ConfigReader.swift:21-27).
**Proposal:** Build one `Pill` view (params: text, optional systemImage, tint, filled-vs-outlined) in SurfaceChrome.swift and migrate all six call sites. Fix the wording drift by changing OptionListView.swift:1907 to render `option.state.displayName` instead of the hand-typed literal.

### DS-5 [P2/refinement/M] accentColor is spent on ~7 unrelated roles (state, selection, provenance, decoration) at once, against research's 'one accent axis' rule
**Evidence:** accentColor drives: the active auto-reload chip (GhosttyConfigEditorApp.swift:101,460), the 'Customized' pill/text (OptionListView.swift:425-426,1907), the current-theme label/selection-ring/pill (ThemeBrowserView.swift:130,180,213-214), the selected font radio dot (OptionListView.swift:1574), the selected preset-swatch ring (OptionListView.swift:908), the search-match provenance label (GlobalFindView.swift:182), and Welcome decoration/CTA (WelcomeView.swift:71,108,137,148). research-context.md calls for 'ONE accent axis applied narrowly to focal UI'; the app spends it on state + selection + provenance + decoration simultaneously (Obs 9/36).
**Proposal:** Reserve accentColor for the single most focal cue per surface (current-item/selection and the primary CTA). Give 'customized/non-default state' its own non-accent tint — precedent already exists in-repo (`.orange` for 'Replaces a default' at KeybindEditorView.swift:574 and 'not using yet' at OptionListView.swift:1911) — so 'this differs from default' is legible apart from 'this is currently selected.'

### DS-6 [P3/refinement/S] The active auto-reload toolbar chip dyes its ENTIRE label accent, reading as a hyperlink and violating the chip's own 'tint the glyph only' contract
**Evidence:** ToolbarChip's doc comment states 'tint colors only the leading glyph' (GhosttyConfigEditorApp.swift:88-89), but the actionable branch applies `content.foregroundStyle(isActive ? Color.accentColor : .primary)` to the whole label (line 101). autoReloadChip is `isActive` whenever auto-reload is on (lines 457-463), so 'Auto-reload: On' renders entirely accent-blue — Obs 2: it 'reads like a hyperlink,' and it is the only state cue that drains when the window is unfocused. (The identity chip is already plain `.secondary` text with no border/glyph, so the two are structurally distinct; this is the real defect, not indistinguishability.)
**Proposal:** Apply accent only to the leading `arrow.clockwise` glyph (the tint the component already documents) and keep the label text `.primary`. On/off state stays conveyed by the persistent 'On'/'Off' text plus the glyph tint, so the chip reads as a status control focused or not and stops looking like a link next to the identity label.

### DS-7 [P2/fix/S] `role: .destructive` on Form-row 'Reset All to Defaults…' buttons renders no destructive tint, contradicting the declared semantic role
**Evidence:** OptionListView.swift:138 and SettingsView.swift:69 both declare `Button("Reset All to Defaults…", role: .destructive)` inside a `Form { Section }` with `.formStyle(.grouped)`, but the plain Form-row button style ignores the role on macOS — Obs 41: it is 'styled identically to Copy Full Config (no destructive tint).' The same `.destructive` role DOES render red on the confirmationDialog buttons (OptionListView.swift:51 / SettingsView.swift:45), confirming the row-level annotation is silently dropped.
**Proposal:** Add an explicit `.foregroundStyle(.red)` (or a shared `DestructiveRowButton` wrapper) to both Form-row reset buttons so the row visually warns before the confirmation dialog appears, and reuse the wrapper for any future destructive row button.

### DS-8 [P3/refinement/S] Typography scale has no reserved hero step — Welcome's title uses the same `.title2` token as every routine surface header
**Evidence:** SurfaceHeader (SurfaceChrome.swift:42) uses `.font(.title2.weight(.semibold))` for every routine header ('Themes', etc.); WelcomeView.swift:74 uses the identical token for the sole once-per-install identity screen. Welcome already carries an icon hero (WelcomeView.swift:69, `gearshape.2.fill` at `.title`), so the gap is specifically typographic: the scale tops out at the routine chrome step, so the identity moment cannot visually outrank anything (Obs 43).
**Proposal:** Define a small semantic type scale (e.g. `heroTitle = .largeTitle.weight(.bold)` reserved for Welcome/empty-hero, `surfaceTitle = .title2.weight(.semibold)` for routine headers) and give Welcome its own top rung, without touching every other surface.

### DS-9 [P2/fix/S] The mid-row 'Customized' pill has no layout-priority protection, so the label/subtitle column wraps and truncates when it appears
**Evidence:** OptionRow.body (OptionListView.swift:384-414) is `HStack { VStack(title+subtitle); if setNonDefault { customizedAffordance }; Spacer(minLength:12); editor; infoButton }`. `grep layoutPriority` across OptionListView.swift returns zero hits, so when the pill+reset cluster is present it competes for width against the fixed editor/info controls and the label VStack yields — Obs 9: 'Confirm before closing' and 'Reopen windows on launch' wrap to two lines and their subtitles truncate. (This is the tactical fix; the strategic version is the metadata-consolidation finding.)
**Proposal:** Give the label `VStack` `.layoutPriority(1)` (and/or `.fixedSize` the pill+reset cluster) so the title/subtitle column is never the one sacrificed for the pill's width.

### DS-10 [P2/fix/M] The per-row feedback and the shared SurfaceFeedbackBar duplicate the same Saving…/Saved/Undo content with drifted fonts — proof there is no single feedback component
**Evidence:** OptionRow.feedback (OptionListView.swift:492-537) and SurfaceFeedbackBar (SurfaceChrome.swift:158-206) are structurally copy-pasted switches over `applyState`, but the type tokens drift: 'Saving…' is `.caption2` (OptionListView.swift:501) vs `.caption` (SurfaceChrome.swift:166); the Undo button is `.caption2` (:518) vs `.caption` (:178); the failed message is `.caption2` (:525) vs `.caption` (:195). Two feedback geometries for the same Saved/Undo semantics (Obs 44).
**Proposal:** Extract the state→content mapping (icon, color, copy for applying/succeeded/failed) into one shared view that both call sites parameterize only by placement (inline-under-row vs bottom-bar), removing the drift risk permanently rather than patching individual font mismatches.

### DS-11 [P2/feature/L] Each option row runs three competing metadata systems (trailing ⓘ, mid-row pill+reset, popover badges) that fight the controls for space
**Evidence:** OptionRow.body shows per row: title+subtitle, an always-visible pill+reset cluster (customizedAffordance, OptionListView.swift:419-441) and a trailing infoButton (469-487); the popover then adds a 4th badge layer (category/type/repeatable/state, 1892-1898). Obs 15 ('12 identical ⓘ per screen; right edge is a column of noise') and Obs 48 ('three metadata systems competing with the actual controls') are the live symptom. This revisits the pass-1 B5 mid-row-pill decision — justified by that NEW friction evidence, flagged explicitly.
**Proposal:** Consolidate into one trailing accessory per row: fold the reset action into the ⓘ popover (Obs 18 already asks for a popover Reset) and replace the always-visible pill with a lighter always-present cue (a small filled state dot or left-edge rule reusing the new non-accent state tint) that doesn't compete for horizontal width. Turns three systems into two (control column + one accessory). Supersedes the tactical layoutPriority fix.

### DS-12 [P3/refinement/S] GlobalFindView re-declares SurfaceHeader's search-field recipe verbatim instead of composing the shared component
**Evidence:** GlobalFindView.swift:74-76 (`.padding(.horizontal,8).padding(.vertical,6).background(Color.secondary.opacity(0.10), in: RoundedRectangle(cornerRadius: 8))`) is identical to SurfaceChrome.swift:79-81's `searchField`, written independently rather than reused. They stay in sync only by luck — the same way the two feedback implementations already drifted their 'Saving…' font (separate finding).
**Proposal:** Extract `SurfaceHeader.searchField` into a standalone reusable view (or a static builder) and have GlobalFindView call it, instead of re-declaring the HStack/padding/background recipe.

### DS-13 [P3/fix/S] Option-info popover shows the raw enum rawValue ("enumeration") when the kit already ships a friendly displayName ("Choice")
**Evidence:** OptionListView.swift:1894 renders `Badge(text: option.option.valueType.rawValue, systemImage: "tag")`. OptionCatalog.swift:19-28 defines `OptionValueType.displayName` returning "On/off"/"Number"/"Color"/"Choice"/"Text" (and `nil` for `.unknown`) for exactly this purpose. Obs 16: 'Type chip shows raw "enumeration" (kit has displayName "Choice" — not used here).'
**Proposal:** Change line 1894 to `Badge(text: option.option.valueType.displayName ?? option.option.valueType.rawValue, systemImage: "tag")` — the `??` fallback is required because `displayName` is optional for `.unknown`. Good candidate for a lint/guard-test that flags any `.rawValue` rendered directly to a badge/label.

### DS-14 [P3/refinement/M] KeyRecorderView is a parallel hand-drawn AppKit styling system disconnected from the app's SwiftUI pill idiom, so a shortcut capsule reads as an empty text field
**Evidence:** KeyRecorderView.swift:248-296 `draw(_:)` hand-draws its capsule with `NSBezierPath(roundedRect:xRadius:6,yRadius:6)`, `NSColor.controlBackgroundColor`/`.separatorColor`/`.controlAccentColor` and `NSFont.systemFont` — none of it references the app's radius/tint tokens or its Capsule pill language. Obs 34: 'Recorder capsules look like text fields; no per-row record affordance/icon' — a direct consequence of the bezel matching an NSTextField, not a shortcut chip.
**Proposal:** Give the recorder a distinguishing cue independent of focus (a small record-dot glyph, or a dashed vs solid stroke) so it reads as 'a shortcut capsule' before it's focused; when the design-token constants land (separate finding), mirror the radius/tint values as `CGFloat`/`NSColor` constants importable here so this view stops being a second source of truth.

## Interaction feel & motion (MO)

### MO-1 [P1/fix/M] First click after dismissing a Menu/popover is swallowed across many controls, so apply/open actions silently need two clicks
**Evidence:** Obs 25 saw this live on the theme ⋯ pairing Menu (ThemeBrowserView.swift:287-306), the font-picker popover, and popovers generally — the click right after dismissal does nothing; a second click acts. Verified there is no self-inflicted dismiss-scrim: grep finds a single onTapGesture in the whole app (WelcomeView.swift:20, the unrelated welcome scrim), and ~11 .popover/Menu sites (ThemeBrowserView:288; OptionListView:484,671,741,1320,1653,1790; SurfaceChrome:91; KeybindEditorView:345,368,369) all rely on SwiftUI's built-in dismiss — so this is a SwiftUI popover/Menu dismiss-vs-hit-testing race, not app code eating events on purpose.
**Proposal:** Spike to confirm the SwiftUI dismiss-timing root cause, then apply a targeted AppKit mitigation to the 2-3 highest-traffic offenders first (theme ⋯ menu, the info popover): an NSEvent local mouse-down monitor that force-closes the open popover/menu synchronously before the triggering click is dispatched, so one click both dismisses and acts. Precedent for dropping to AppKit already exists (KeyRecorderView.swift owns its own NSEvent monitor). Scope to the highest-frequency sites — do not blanket-rewrite every popover/Menu.

### MO-2 [P2/refinement/M] Per-row Options 'Saved · Undo' feedback snaps in and never times out, leaving a permanent ~35px layout shift under the app's most frequent action
**Evidence:** Obs 14/46. OptionRow.feedback (OptionListView.swift:492-537) renders Saving/Saved/error via a plain switch with no .transition. applyEdit (AppModel.swift:331-361) assigns applyState directly, never inside withAnimation. resetApplyState() (AppModel.swift:380-383) is called only on a selection change (App:402), reloadFromDisk (AppModel:395), and before a new keybind edit (KeybindEditorView:543) — verified by grep; no caller clears it after an Options edit, so '✓ Saved · Undo' sits under that row indefinitely, reserving vertical space until the user navigates away or edits another option.
**Proposal:** (a) Wrap only the .succeeded/.idle assignments in withAnimation(reduceMotion ? nil : .easeOut(duration: ~0.18)) and add .transition(.opacity.combined(with: .move(edge: .top))) to the feedback view (leave .applying unanimated so it stays instantaneous). (b) After .succeeded shows ~2.5s untouched, collapse it to a small static checkmark using the self-resetting Task.sleep pattern already proven by the Copy-snippet button (OptionListView.swift:2025-2034). This is view-local timing on an already-committed write — it adds no debounce/optimistic-UI and does not touch the un-serialized apply race (AppModel:331); ⌘Z remains the durable undo path after the row collapses.

### MO-3 [P2/refinement/M] The theme surface — Ghostty's most brand-relevant, highest-traffic interaction — has no hover cue and the applied row snaps rather than 'lands'
**Evidence:** Obs 22/45. ThemeRow (ThemeBrowserView.swift:141-167) has zero .onHover (grep-confirmed) while a click applies the theme and live-reloads instantly (line 154-159). The accent border marking the applied row (strokeBorder isCurrent, lines 178-183) and the 'Current' pill (line 189; currentPill 207-215) have no .animation/.transition, so the only on-row signal is an instant color snap; the 'Saved' text lives in a disconnected bottom bar (Obs 23, SurfaceChrome.swift:154-217).
**Proposal:** Primary value: animate the isCurrent-dependent border and pill with withAnimation(reduceMotion ? nil : .easeOut(duration: 0.25), value: isCurrent) so the applied row visibly settles instead of snapping — on-brand for Ghostty's color-fidelity ethos. Secondary: give rowLabel a hairline hover tint (~5% opacity, no movement) drawn from the shared strengthen-on-hover style proposed separately, so browsing hundreds of rows before an undoable apply has a 'this is what I'm about to apply' cue. Keep hover coordinated with the app-wide hover style rather than hand-rolling a one-off here.

### MO-4 [P2/fix/M] Advanced disclosure is authored with a 0.15s withAnimation but the rows snap in — an ambient withAnimation doesn't animate conditional row insertion inside a grouped Form Section on macOS
**Evidence:** Obs 26 ('Expansion has no animation; rows just appear') directly contradicts the authored intent: advancedHeader (OptionListView.swift:345-366) wraps advancedExpanded.toggle() in withAnimation(.easeInOut(duration: 0.15)) at line 348, but the row content swap is a bare `if advancedExpanded { rows(advanced) }` (lines 321-326) inside a grouped Form Section with no .transition on the rows — so nothing interpolates.
**Proposal:** Add an explicit .transition(.opacity) to the inserted advanced rows (opacity survives Form/List insertion more reliably than move/height on macOS) and verify in the packaged app per project convention (view-only change → launch-verify, not unit test). If opacity still doesn't animate inside the grouped Form Section, fall back to always rendering the advanced rows and collapsing them behind .frame(height: 0).clipped() + opacity fade instead of the conditional.

### MO-5 [P2/feature/L] No hover-state system: icon buttons, the disclosure header, Welcome cards, and Find rows never strengthen on hover
**Evidence:** Obs 45; onHover count is 0 across all of Sources/ (grep). The info button (OptionListView.swift:469-487), reset glyph (419-441), Advanced-disclosure header (345-366), Welcome jump-in cards (WelcomeView.swift:100-127), and Find result rows (GlobalFindView.swift:121-158) are all always-visible plain-styled buttons with no hover feedback, so a pointer crossing the ~12-icon trailing column (Obs 15) or the Welcome cards gets no cue about what will receive a click.
**Proposal:** Add one small reusable ButtonStyle (e.g. HoverAffordanceButtonStyle) that uses .onHover to lift background opacity by a hairline (~4-6%, no movement/scale) on icon-sized controls, with the same restrained treatment for card/row-sized ones. It must strengthen controls already visible at rest — never reveal otherwise-invisible ones (research-context flags System Settings' hover-only affordance as the anti-pattern to avoid). Per HIG, hover-tint is not decorative motion and can stay ungated by Reduce Motion, unlike the entrance/disclosure animations elsewhere in this pass.

### MO-6 [P3/refinement/M] The Customized and Current pills appear/disappear abruptly, and five near-identical capsule pills are hand-rolled with no shared component to animate
**Evidence:** The two pills that genuinely pop in/out on a state change — customizedAffordance (OptionListView.swift:404-406, defined 419-441) and currentPill (ThemeBrowserView.swift:189, defined 207-215) — have zero .transition (Obs 46). The keybind origin badge (KeybindEditorView.swift:549) and Find categoryPill (GlobalFindView.swift:164-171) are persistent-per-row (their content changes, not their presence), so they benefit less. Separately, the private Badge component already exists and is used only for the info-popover chips (OptionListView.swift:1893-1911) while the four pills above are each styled independently with near-identical accentColor/secondary-in-Capsule code (research-context).
**Proposal:** Extract Badge from private into a small shared type and retarget the hand-rolled pills to it (preserving each look), consolidating the ~5 duplicated capsule stylings. Give it one .transition(.scale(scale: 0.9).combined(with: .opacity)) + reduce-motion fallback so the genuinely appearing/disappearing pills (Customized, Current) get consistent motion for free. Framing: the consistency/consolidation is the main win here; the motion payoff is limited to the two state-triggered pills.

### MO-7 [P3/feature/M] KeyRecorder (hand-drawn AppKit view) has no hover and flips into recording mode with an instant unanimated redraw, on the app's densest surface
**Evidence:** No NSTrackingArea/mouseEntered/NSAnimationContext anywhere in KeyRecorderView.swift (grep-confirmed). isRecording's didSet (lines 49-58) only sets needsDisplay = true and re-notes the focus-ring mask; the border/fill flip that marks 'you are now recording' (draw(_:) lines 251-254, controlAccentColor vs controlBackgroundColor/separatorColor) is an instant redraw. This mode-change can occur on up to 136 rows (Obs 31-35).
**Proposal:** Add an NSTrackingArea (.mouseEnteredAndExited, .activeInKeyWindow) so the recorder gets the same hover-strengthen treatment as its SwiftUI siblings, and animate the recording-state fill/border swap via a short layer-backed CABasicAnimation / NSAnimationContext fade (~120ms) instead of the instant needsDisplay redraw, gated on NSWorkspace.shared.accessibilityDisplayShouldReduceMotion to mirror the SwiftUI-side reduce-motion behavior. Contained addition to a file that already owns its drawing/lifecycle; note layer-backing the NSView is the non-trivial part of the effort.

### MO-8 [P3/refinement/S] Global Find calls itself an overlay but opens/closes with a hard cut identical to ordinary navigation
**Evidence:** GlobalFindView's doc comment literally names it 'The global Find overlay' (GlobalFindView.swift:4-9), but mainColumn renders it as `if model.isFinding { GlobalFindView() } else { switch … }` with no .transition/.animation (GhosttyConfigEditorApp.swift:348-365) — ⌘F and Esc produce an instant content-pane swap indistinguishable from clicking a different sidebar row.
**Proposal:** Add a brief reduce-motion-gated fade to just this toggle: .animation(reduceMotion ? nil : .easeOut(duration: 0.15), value: model.isFinding) on the Group in mainColumn. Because the trigger is keyed to model.isFinding, ordinary category switches (model.selection) stay instant like native System Settings — only summoning/dismissing Find reads as a temporary layer rather than a navigation destination.

### MO-9 [P3/feature/S] No shared motion vocabulary — every animation duration/curve is invented ad hoc
**Evidence:** The only three explicit animation sites each hard-code their own literal: withAnimation(.easeInOut(duration: 0.15)) (OptionListView.swift:348), .animation(reduceMotion ? nil : .easeInOut(duration: 0.2), value:) (GhosttyConfigEditorApp.swift:318), plus unrelated 400ms/350ms debounce timers (OptionListView.swift:1173, 1748). RowMetrics (SurfaceChrome.swift:12-19) already proves the codebase's own named-constant pattern for exactly this kind of shared rhythm, but nothing analogous exists for motion.
**Proposal:** Add a small MotionSystem enum next to RowMetrics in SurfaceChrome.swift (e.g. static let quickFade: Animation = .easeOut(duration: 0.15); static let settle: Animation = .easeOut(duration: 0.25)) so every new transition in this pass — feedback collapse, theme-apply landing, disclosure, Find fade — draws from one restrained named vocabulary instead of proliferating one-off durations. Enabler for the other motion findings; low value on its own.

## Controls & value presentation (CV)

### CV-1 [P1/fix/M] Enum-value raw-token leak has no fallback humanizer — only 4 of ~300 options are curated
**Evidence:** NumericSpec.swift:108-110 `EnumValueLabels.label` returns `labels[option]?[value] ?? value` — raw value, no humanization; enum-value-labels.json curates only 4 options (link-previews, confirm-close-surface, custom-shader-animation, macos-option-as-alt). enumChoices (ConfigReader.swift:103,115) feeds every picker row through `option.enumValueLabel($0)`, so uncurated values render raw. `cursor-style` and `window-save-state` are both on the first Recommended screen (recommended-settings.json:15,18) and render 'bar'/'always' verbatim (Obs 10). `cursor-style-blink` is a tri-state absent from booleanishOptions (CatalogParser.swift:70-77) so it renders as an .enumeration Picker whose 'true' shows literally (Obs 37). The same raw string surfaces in OptionInfoPopover's 'Your value'/'Default' rows (OptionListView.swift:1964-1975), which read userValues/defaultValue with no label pass (Obs 18). By contrast LabelCatalog.humanize and ActionLabelCatalog both HAVE deterministic humanizers — enum values are the one leaf without one.
**Proposal:** Add a deterministic fallback to `EnumValueLabels.label`, mirroring `LabelCatalog.humanize`: (1) map `true`→'On'/`false`→'Off' first (curated overrides like link-previews still win because they're looked up before the fallback); (2) humanize remaining snake/kebab tokens via LabelCatalog.humanize (`block_hollow`→'Block (hollow)', `left`→'Left'). Keep the 4 curated overrides for phrasings the humanizer can't produce (`osc8`→'Only OSC 8 hyperlinks'). For the popover's 'Your value'/'Default' rows, apply the enum label ONLY when the option is `.enumeration`/boolean-ish so hex/path/number values aren't mangled. Add an orphan/coverage test asserting no catalog enum value renders as its own raw lowercase token.

### CV-2 [P2/refinement/M] Opacity-family sliders show a bare 0–1 decimal with no unit, and 'transparency' label points opposite the stored value
**Evidence:** OptionListView.swift:1027-1042 `sliderEditor` renders `Text(numberString(spec.clamp(live), decimals:))` with no unit/percent suffix and no anchor captions, for all six min:0(0.15)/max:1 sliders in numeric-specs.json:4-9 (background-opacity 'Window transparency', unfocused-split-opacity 'Inactive split dimming', background-image-opacity, cursor-opacity, faint-opacity, bell-audio-volume). fieldEditor (:1073-1075) already reads `spec.unit`, but sliderEditor never does — the plumbing exists for one style and is missing for the other. background-opacity default 1 renders as literal '1' under a 'transparency' label (Obs 11); the number is technically the opacity value but reads backwards from the label.
**Proposal:** Add a percent display mode to NumericSpec (a Style or a `unit:"%"` convention sliderEditor honors) so 0–1 sliders render `Int(round(value*100))%` — fixes all six at once. Independently resolve the directional ambiguity by relabeling 'Window transparency'→'Window opacity' (matches Ghostty's key semantics, no math change), or add fixed 'More transparent'/'More opaque' anchor captions flanking the slider.

### CV-3 [P2/fix/S] Font picker popover title shows the raw catalog key ('font-family'), not the option's displayTitle
**Evidence:** OptionListView.swift:1376 `Text(option.option.name)` in FontFamilyEditor.picker renders the raw key; every other title in the file uses `option.option.displayTitle` (e.g. :749, :878, :1883). option-labels.json:4 already curates font-family→'Font', so it's a one-line swap; bold/italic variants fall back to the humanizer ('Font family bold'), still better than the raw key (Obs 30).
**Proposal:** Change OptionListView.swift:1376 from `Text(option.option.name)` to `Text(option.option.displayTitle)`.

### CV-4 [P2/fix/S] Info popover's 'type' chip shows the raw enum case name instead of the kit's already-built friendly label
**Evidence:** OptionListView.swift:1894 `Badge(text: option.option.valueType.rawValue, systemImage: "tag")` renders 'enumeration'/'boolean'/'string' verbatim (Obs 16). ToolbarChip (GhosttyConfigEditorApp.swift:90-128) renders the non-interactive identityChip (plain .secondary text, no border, 447-450) and the clickable stateful autoReloadChip (.buttonStyle(.bordered), accent-tinted only when on, 456-469) at the same .caption size (content :126) with the same HStack spacing, immediately adjacent. ### CB-11 [P3/fix/S] Color swatch border is a fixed low-contrast hairline that visually disappears against dark values
**Evidence:** Obs 27. **Proposal:** Replace the .separator stroke with a border guaranteed to contrast against both extremes — a fixed mid-gray (e.g. Color(white: 0.5).opacity(0.6)) or a double inner/outer light+dark hairline, the way system color wells guard this case. Apply to both the row swatch (:836) and the popover's larger swatch (:876) for consistency.

## Completeness-critic gaps (GAP)

### GAP-1 [P1] Light-mode has never been driven, yet the entire pass-2 visual system is being tuned in dark-only — the new tint/overlay/mono/material tokens are luminance-directional
**Why:** Obs 49 states light mode was NOT verified this session (system was dark); all 50 observations and every lens proposal are dark-grounded. Pass-2 introduces a whole layer of luminance-sensitive tokens: the proposed shared subtle-fill/accent-fill scale, hover-strengthen (+4-6% opacity), the swatch double-stroke ring, mono-as-accent chips, and the 'one systematic overlay rule' for chrome-over-theme-preview all read opposite in light mode. Concrete existing landmines that pass-2 stacks onto: WelcomeView.swift:123 jump-in cards fill Color.primary.opacity(0.04) — near-black at 4% on a light material card is nearly invisible in light mode (fine in dark as 4% white); WelcomeView.swift:148 accentColor.opacity(0.08) CTA and :97 green.opacity(0.10) strip; the only .regularMaterial (WelcomeView.swift:61) renders very differently light vs dark. The proposed mini-terminal theme preview and its chrome overlay are defined against dark backgrounds only. Half the user base may run light mode and no one has looked once.
**Proposal:** Add a mandatory light-mode verification pass to the plan (repackage, toggle System Appearance to Light, drive Welcome, Themes, an option category with swatches, Keybinds, Problems, Settings) as an explicit gate before sign-off. Make it a RULE, not a one-time check: every new tint token in the DesignTokens layer must be defined luminance-relative (semantic .secondary/.quaternary fills or an explicit light/dark pair via colorScheme), never a fixed dark-assumed alpha over an assumed-dark surface; the swatch ring and terminal-preview overlay must follow the research-grounded 'light overlay on dark, dark on light' rule; verify the WelcomeView material card and the 0.04/0.08/0.10 fills in both appearances and re-tune. Cross-references Obs 27 (swatch invisibility) which is itself appearance-asymmetric.

### GAP-2 [P2] Keyboard-only navigation and focus order after the restructures were never audited — the per-surface search field is not focusable-on-entry and the new hover affordances have no keyboard equivalent
**Why:** No lens covered keyboard-only operation or focus order; the walkthrough was click/hover-driven. Verified: the per-surface filter is a plain TextField with no @FocusState (SurfaceChrome.swift:70) — entering Themes/Options/Keybinds never focuses the local filter, so a keyboard user must mouse-click it, while ⌘F Find DOES auto-focus (GlobalFindView.swift:16,65). research-context.md explicitly names 'keyboard focus in search' and 'keyboard-nav gaps' as System-Settings idioms to beat. There is no defaultFocus/.searchable anywhere. Worse for pass-2: the interaction and themes lenses propose an app-wide hover-strengthen system on icon buttons, disclosure headers, theme rows and cards — hover-only strengthening is invisible to keyboard/VoiceOver users unless a matching visible focus state ships in lockstep (the same anti-pattern research warns about, inverted). Nobody verified tab order through the consolidated trailing accessory, the Advanced disclosure, or focus behavior after a theme applies or the disclosure expands.
**Proposal:** Add a keyboard-only navigation unit/gate: (a) give the per-surface SurfaceHeader search field a @FocusState and either focus-on-surface-entry or a ⌘F-scoped-to-local-filter binding so search is reachable without the mouse; (b) require that every hover-strengthen affordance introduced in this pass also defines a visible keyboard focus state (focus ring / same tint on isFocused), verified by tabbing; (c) add a tab-order verification checklist for each restructured surface (option row trailing accessory, disclosure header, theme row controls) and confirm focus is sensibly placed after Advanced-expand and after theme-apply.

### GAP-3 [P2] The structural pass-2 changes rewrite the accessibility tree that pass-1 H1-H3 hardened, but no VoiceOver-regression gate is attached to them (only net-new a11y was in scope)
**Why:** Pass-1 shipped option-list / keybind / Problems a11y (H1-H3, commits ee6d9a5/38912ab/4e0452d) including per-control labels and an explicit theme accessibilityLabel (ThemeBrowserView.swift:308) and the recorder de-combine. Several pass-2 proposals REPLACE or MOVE the very nodes those labels hang on, risking silent regressions no lens flagged: the themes lens replaces the swatch-grid preview body (whose accessibilityLabel is hand-built at :308) with a mini-terminal mockup — if the label isn't rebuilt, VoiceOver users lose the theme description; the design-system/IA lenses fold the per-row Reset into the ⓘ popover and suppress the always-visible 'Customized'/'Default' pills — removing a directly focusable, H1-labeled reset button and changing the focus path; mono-as-accent demotes raw ids to .help() which changes what VoiceOver reads. Only one a11y regression (the size-caption accessibilityHidden) was caught, incidentally, by the content lens — there is no cross-cutting requirement to re-run VoiceOver on each consolidation.
**Proposal:** Attach an explicit VoiceOver-regression gate to every unit that replaces or relocates an interactive node or its label: re-verify with VoiceOver that (1) the new terminal-preview cell still announces the theme name/appearance, (2) reset/customized state remains announced and actionable after folding into the popover or replacing the pill with a state dot, (3) demoting ids to .help() doesn't drop them from the a11y value where power-users relied on them. Add these to the H-phase verification list rather than treating pass-2 as visual-only.

### GAP-4 [P2] Window-resize behavior across the full min-to-max range is unaudited — the clipped-Settings-row bug is one instance of a class only seen at the default size
**Why:** The window is drag-resizable (maxSize set at App:77; minWidth 660 / minHeight 520 at :179; no windowResizability(.contentSize)), so users will resize, but every observation (Obs 4/5/9) was at the single snug default 780x660. The IA lens fixes the clipped Settings row via a ScrollViewReader, but only that row — it doesn't audit the class. At minHeight 520 the 17 sidebar rows + 3 section headers cannot all fit, so the same off-screen problem recurs for Problems/Settings regardless of the scroll fix; the ~35px inline feedback shift (Obs 14) eats more of a short window. On width: the themes lens proposes a 2-column LazyVGrid of ~300pt cards, but at minWidth 660 minus the sidebar (~220pt) the content region is ~440pt — two 300pt cards do NOT fit, and at maxWidth 900 the 640pt cap centers and strands voids the grid must fill gracefully. No proposal states how the grid column count responds to available width, or that surfaces must be verified at the extremes.
**Proposal:** Add a resize-verification gate covering minWidth 660, minHeight 520, and maxWidth 900 for every surface, and specify column-count responsiveness for the proposed theme grid (adaptive GridItem so it degrades to 1 column below a width threshold rather than clipping). Confirm the sidebar ScrollViewReader fix (IA lens) actually reveals off-screen Status rows at 520px, and that inline feedback doesn't strand content at min height. ### GAP-5 [P2] Nobody costed the richer previews or the All/Dark/Light filter at 463 themes — the classification pass and heavier per-row mockup have no performance budget or cancellation contract
**Why:** The themes/character lenses propose (a) replacing the 16-rect+‘Aa’ swatch with a multi-layer mini-terminal mockup per row and (b) a 2-column grid rendering ~2x visible cards and (c) an All/Dark/Light/Favorites filter. None address rendering/parse cost. Today ensureColors (AppModel.swift:730) is lazy per-row via .onAppear, off-main, cached in themeColors, and guarded so a re-bootstrap cancels in-flight loads (cancelInFlightColorLoads). The appearance filter fundamentally breaks that laziness: classifying 463 themes by ThemeColors.appearance requires reading+parsing ALL 463 theme files up front (the lens hand-waves a 'batch background read'), which must (1) stay off the main thread, (2) show a determinate 'Classifying N…' progress affordance, and (3) reuse the exact provider-identity/cancellation guard so a mid-batch re-bootstrap can't write stale colors into the new model (institutional constraint). **Proposal:** Give the themes units an explicit performance contract: keep LazyVGrid/List lazy (never eager VStack), keep preview rendering derived only from already-cached ThemeColors, and specify the batch-classify path (background task, cancellable via the existing colorTasks/provider-guard pattern, determinate progress, results memoized so re-selecting a filter is instant). ### GAP-6 [P2] First-run/no-config and the four non-ready environment states got pass-1 plumbing but no pass-2 character, and were never re-driven this session
**Why:** Obs 50 confirms the walkthrough ran against a HEALTHY environment (binary present, config auto-created); the four non-ready screens — loading, notFound, unsupported, failed (App:279-312, recovery actions :327-339) — were shipped in pass-1 G1 but never re-observed in pass-2, so none of them receive the character/typography/motion/light-mode treatment the plan is applying to the happy path. ### GAP-7 [P2] Cross-proposal collisions: the design-token layer must be primitive-bridged to the AppKit KeyRecorder, and the hover system must be sequenced with the swallowed-first-click NSEvent mitigation and a hover-capable verification method
**Why:** The critic's mandate is interactions BETWEEN proposals. Two concrete collisions no single lens owns: (1) The proposed DesignTokens layer is SwiftUI (Color/CGFloat), but KeyRecorderView is a hand-drawn AppKit NSView using NSColor/NSBezierPath with its own radius 6 and reduce-motion via NSWorkspace.accessibilityDisplayShouldReduceMotion (a different API than SwiftUI's @Environment(\.accessibilityReduceMotion)). If tokens ship SwiftUI-only, the recorder stays a second source of truth the instant it's restyled (multiple findings want it restyled), and its proposed CABasicAnimation reduce-motion gate won't share the SwiftUI path. (2) The app-wide hover-strengthen system (interaction lens, L) and the swallowed-first-click fix (interaction/themes P1, an NSEvent local mouse-down monitor) both touch the same high-traffic controls; an event monitor that force-closes popovers must not swallow or race the NSTrackingArea hover events the recorder/rows are gaining. And the project's verification is click-driven (memory: click-only tier), which cannot exercise hover at all — so a hover-heavy pass has no stated verification method.
**Proposal:** Specify tokens as framework-neutral primitives (CGFloat radii/spacing + a hex/NSColor bridge) that BOTH the SwiftUI views and KeyRecorderView import, so 'single source of truth' actually holds across the AppKit boundary, and define one reduce-motion helper that maps to both the SwiftUI env and the NSWorkspace flag. Sequence the swallowed-click fix before or alongside the hover system and verify they coexist on the shared controls. Add a hover-verification method to the plan (mouse-move probes via the full-tier computer-use path, not click-only) since the standard click workflow can't see hover regressions.

### GAP-8 [P2] The unserialized applyEdit race must become an explicit prerequisite gate for the pass-2 units that raise apply frequency or add timing around apply — the lenses each deferred it per-unit instead
**Why:** Project memory (deferred-apply-serialization) and research-context.md both flag AppModel.applyEdit (AppModel.swift:331) as having NO in-flight serialization (rapid edits clobber / go stale-on-disk), with PaletteEditor's working-copy+debounced-commit as the in-repo template. **Proposal:** Elevate the applyEdit serialization fix from a per-unit deferral to a named prerequisite in the phase dependency graph, gating the theme-apply-motion, hover-preview, feedback-timer, and batch-classify units behind it (or requiring each to route through the PaletteEditor working-copy+debounced-commit pattern). ### GAP-9 [P3] New pass-2 copy and fixed-size elements aren't run through Dynamic Type / long-string fragility checks, and new strings bypass the catalog discipline
**Why:** Pass-1 H3 shipped Dynamic Type support, but pass-2 ADDS copy and fixed-metric elements that no lens stress-tested at accessibility text sizes or against overflow: slider endpoint captions ('Opaque'/'Transparent', 'More transparent'/'More opaque') that flank an already width-constrained control (Obs 9 rows wrap at default size), the terminal-preview fixed sample string ('AaBbGg 0123 -> => !='), the 'Classifying N…' batch label, a reserved hero title step, and the consolidated Pill/mono chips whose fixedSize competes with the label column.
